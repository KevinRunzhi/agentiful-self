## 1. 产品概述

### 1.1 产品定位与目标

**背景**：企业采用多种 LLMOps/Agent/Workflow 平台（Dify、Coze、n8n 等），缺乏统一入口与企业级治理能力。

**定位**：AgentifUI 是面向企业的 AI 应用统一管理平台，提供统一前端入口与企业级治理能力，不替代后端编排平台。

**核心目标**：

- **统一体验**
  - 描述：屏蔽底层差异，提供一致 AI 交互体验
  - 价值承载者：普通用户

- **企业管控**
  - 描述：可控、安全、合规的 AI 使用环境
  - 价值承载者：管理员/IT

- **治理可视**
  - 描述：清晰的权限、配额、成本统计与审计
  - 价值承载者：管理者/合规人员

- **架构解耦**
  - 描述：作为中间层解耦前端与 AI 引擎
  - 价值承载者：技术团队

### 1.2 目标用户

- **普通用户（User）**
  - 核心目标：高效完成任务
  - 主要行为：对话、运行 Agent/Workflow、查看历史
  - 关注点：体验一致、上下文连续

- **群组管理者（Manager）**
  - 核心目标：团队效率提升
  - 主要行为：成员管理、应用授权、配额查看
  - 关注点：权限边界、成本控制

- **平台管理员（Admin）**
  - 核心目标：平台安全稳定
  - 主要行为：认证配置、应用注册、策略设置
  - 关注点：安全、权限、合规
  - 对应 RBAC 角色：**Tenant Admin**（租户级）或 **ROOT ADMIN**（平台级，默认关闭）

- **IT/安全/合规人员**
  - 核心目标：降低风险
  - 主要行为：审计日志、风险分析、异常回溯
  - 关注点：可追溯、合规有效性

**用户旅程**：
- **User**：登录 → 选择应用 → 对话/执行 → 查看结果/历史
- **Manager**：登录 → 成员/应用管理 → 查看配额使用 → 查看团队统计
- **Admin**：登录 → 认证/应用/策略配置 → 审计监控

### 1.3 产品边界

#### In-Scope（v1.0）

- 统一身份认证与 RBAC
- AI 应用交互入口（Chat/Agent/Workflow）
- 对话系统（实时对话、文件、流式、Artifacts、HITL）
- 统一接口网关（兼容 OpenAI API 规范）
- 执行状态与 Trace 追踪
- 权限治理（授权、配额、统计）
- 安全合规（PII 去敏、审计、拦截）
- 平台 API 与 Webhook
- 多语言（中/英）与品牌定制

#### Out-of-Scope

- **编排逻辑**：不实现 Agent/Workflow 的创建/编辑/发布
- **工作流调度**：不替代第三方平台的执行引擎
- **模型训练/推理**：不负责 AI 模型层能力
- **模型评估（Evals）**：归属后端编排层
- **细粒度文档权限**：遵循"应用级授权"，细粒度由后端解决
- **SaaS 商业化**：v1.0 仅做私有化多租户

#### 边界澄清

- 支持 Workflow/Agent **运行与状态展示**，不提供编排能力
- 可提供跳转编排层的深链入口，跳转后需重新登录
- 后端返回的引用来源，前端默认原样展示

### 1.4 核心能力总览

1. 用户与身份管理（多认证、多租户、RBAC）
2. AI 应用统一入口（Chat/Agent/Workflow）
3. 对话与交互系统（实时、文件、流式、Artifacts、HITL）
4. 统一接口网关（OpenAI 兼容、Trace、降级）
5. 执行状态与观测（状态、性能、全链路追踪）
6. 权限与治理（授权、配额、统计、成本）
7. 安全合规与审计（PII、拦截、审计日志）
8. 开放能力（Open API、Webhook）
9. 主题与多语言

## 2. 租户与组织模型

### 2.1 背景与目标

为企业提供数据隔离与组织管理能力，支持多租户部署与团队协作。

### 2.2 核心规则（写死）

**Tenant 定义**：
- Tenant 是平台最高级隔离与治理单元
- Workspace 是 Tenant 的 UI 层别名（1:1 同义），系统内部统一使用 `tenant_id`
- 正文统一使用 **Tenant**，UI 可展示 "Workspace"

**Tenant 隔离范围**（不同 Tenant 强隔离）：
- 用户、群组与角色数据
- 对话内容、执行记录与日志
- 使用统计、配额与治理数据
- 合规策略与审计日志

**Group 定义**：
- Group 是 Tenant 内组织单元，用于成员组织、应用授权、配额分配
- Team 是 Group 的 UI 层别名（1:1 同义）
- 正文统一使用 **Group**

**组织结构**：

- **Tenant**：最高级别的平台治理单元
  - 包含 **Tenant Admin**（租户最高权限）
  - 包含一个或多个 **Group**（群组）
    - 每个 Group 可绑定 **Manager**（群组管理者）
    - 每个 Group 包含一个或多个 **User**（普通成员）

### 2.3 依赖与约束

- 同一用户可属于多个 Group，可同时担任多个 Group 的 Manager
- Manager 权限仅在其被绑定的 Group 内生效
- v1.0 仅支持私有化多租户部署

### 2.4 验收标准

- **租户隔离**：任意租户无法访问其他租户数据（100%隔离）
- **组织层级**：支持 Tenant → Group → User 三级结构
- **多 Group**：单用户可属于 ≥10 个 Group

## 3. 身份认证与账号体系

### 3.1 背景与目标

提供灵活、安全的企业级认证能力，支持多种认证方式与用户生命周期管理。

### 3.2 范围

**In-Scope（v1.0）**：
- 邮箱 + 密码
- 手机号（第三方接口，风控在第三方）
- 第三方账号：Google、GitHub、WeChat
- 企业认证：CAS、OIDC、SAML 2.0、OAuth2
- JIT（Just-in-time）用户创建
- 基于邮箱域名的 SSO 自动识别

**Out-of-Scope**：
- SCIM 用户同步
- 通知类邮件/短信（v2.0+）

> **说明**：事务性邮件（用户邀请、密码重置）属于认证流程，v1.0 支持，不计入「通知系统」。

### 3.3 核心规则（写死）

**认证方式**：管理员可配置启用/关闭各认证方式

**SSO 自动识别**：
- 登录页输入邮箱后，系统根据域名匹配 Tenant SSO 配置
- 匹配成功则列表推荐给用户点击
- 无匹配返回其他登录方式

**账户状态**：活跃 / 暂停 / 待审核 / 已拒绝

**待审核规则**：
- 邮箱/手机号自注册 → 进入待审核
- SSO 用户 → 默认自动激活
- 审批由 Tenant Admin 完成（等待/通过/拒绝）
- 待审核用户可登录，仅可访问个人设置页与状态提示

**用户邀请流程**（管理员主动邀请）：
- Tenant Admin 通过邮箱邀请用户
- 系统发送含激活 Token 的邮件链接
- 用户点击链接 → 设置密码 → **自动激活**（跳过审批）
- 邀请链接有效期：默认 7 天，过期需重新邀请

**密码策略**：
- 默认：≥ 8 位，含大小写 + 数字
- Tenant 可配置更严格策略

**MFA**：
- v1.0 支持 TOTP MFA
- Tenant 可配置：强制 / 可选 / 关闭
- Admin 可为单用户强制启用 MFA（优先级高于 Tenant 级策略）

**用户资料**：头像、昵称、多语言、时区、主题模式

### 3.4 验收标准

- **认证方式**：至少支持 5 种认证方式
- **SSO 识别**：邮箱输入后 ≤ 500ms 返回匹配结果
- **密码强度**：弱密码拒绝率 100%

## 4. 权限与治理模型（RBAC + Scope + 配额）

### 4.1 背景与目标

建立清晰的权限模型，实现细粒度访问控制、配额管理与归因统计。

### 4.2 核心规则（写死）

#### 4.2.1 角色定义（RBAC）

- **ROOT ADMIN**
  - 作用域：Platform
  - 描述：平台运维角色，租户生命周期管理
  - 默认状态：关闭，需显式配置启用
  - **紧急访问（Break-glass）**：仅限运维排障，必须填写事由、**自动发送站内信通知**租户管理员（邮件通知为 v2.0+）并产生最高级别审计记录

- **Tenant Admin**
  - 作用域：Tenant
  - 描述：租户最高治理权限

- **Manager**
  - 作用域：Group
  - 描述：群组管理角色（角色绑定，非独立实体）

- **User**
  - 作用域：User
  - 描述：普通使用者

#### 4.2.2 作用域定义（Scope）

- **Platform**：仅启用 ROOT ADMIN 时使用
- **Tenant**：租户级
- **Group**：群组级
- **User**：用户级

#### 4.2.3 授权模型

**授权模式**：RBAC（角色） + 资源级授权（Grants/ACL）

**授权来源**：

1. **群组 → 应用授权**（默认主路径）
   - 用户通过加入群组获得应用访问权限

2. **用户 → 应用直授**（例外路径，仅 Tenant Admin）
   - 仅 Tenant Admin 可配置，用于绕过群组结构的特殊场景
   - **必须填写**：授权原因、有效期
   - 有效期限制：默认 7 天，最长 90 天
   - 需记录授权原因并纳入审计

3. **Manager 群组内用户级授权**（受限，与用户直授不同）
   - Manager 可在 Tenant Admin 已授权给其 Group 的应用范围内，为群组成员进行**用户级**授权
   - **与用户直授的区别**：依托群组授权范围，不可新增 Tenant 级授权
   - **治理规则**：可选填写授权原因，无有效期限制（跟随群组授权生命周期），纳入审计

**属性约束（ABAC）**：仅作为 Allow/Deny 附加条件，不单独产生授权

**权限优先级**（从高到低）：

1. **显式拒绝（Deny）**：最高优先级，任何显式拒绝都会生效
2. **用户直授（Allow）**：Tenant Admin 直接授权给用户的应用访问权
3. **群组授权 / Manager 用户级授权（Allow，同级）**：通过群组成员资格或 Manager 在群组内为用户分配的授权，两者优先级相同
4. **默认拒绝**：若无任何授权，则拒绝访问

#### 4.2.4 多群组权限合并规则（写死）

- **访问权限**：并集（OR），任一群组授予即可访问
- **配额/扣费归因**：当前工作群组，扣配额群组必须是授予该 App 访问权限的群组之一

#### 4.2.4a 当前工作群组（Active Group Context）

**定义**：用户当前操作归属的群组上下文，用于配额扣减与数据归因。

**选择机制**：
1. **默认**：用户上次使用的群组（本地持久化）
2. **切换入口**：顶部导航栏 Group Switcher
3. **API 参数**：请求 Header 携带 `X-Active-Group-ID`

**约束**：
- 用户仅可选择其所属且被授权访问当前应用的群组
- 切换后立即生效，后续请求归因于新群组
- 若用户仅属于单一群组，隐藏切换器

**智能上下文切换**（体验优化）：
- 用户可见其有权访问的所有应用（并集），不受当前 Context 限制
- 点击应用时：
  - 若当前 Context 已授权：直接进入
  - 若当前 Context 未授权且用户仅属于一个有效群组：自动切换并提示
  - 若属于多个有效群组：弹窗让用户选择 Context

#### 4.2.5 配额模型

**Token 口径**：
- 默认：输入 Token + 输出 Token 之和
- 按模型实际产出（usage-based）归因
- 企业可配置"按请求计量"

**配额作用域（v1.0）**：
- Tenant 级总预算封顶
- Group 级资源分配
- User 级个人边界
- **应用级配额为 v2.0+ Out of Scope**

**配额重置周期**：默认自然月，Tenant 可配置为自然周

**优先级**：细粒度约束优先，不允许突破 Tenant 总上限

#### 4.2.6 内容可见性边界

- **User**：仅可查看自己的对话内容
- **Manager**：可查看团队统计、趋势与用户级指标排行（仅指标，不含内容），**数据范围严格限制为当前 Group 上下文产生的数据**，不可查看用户跨组数据或个人隐私，**不可查看成员对话内容/摘要**
- **Tenant Admin**：可查看成员对话内容，**必须"事由填写 + 审计记录"**

#### 4.2.7 统计归因与报表

**v1.0 统计维度**：Tenant / Group / User / App / Model

**报表指标**：
- 请求量（成功/失败）
- Token（输入/输出/总）
- 成本估算
- 配额消耗与告警
- 错误率、P95 响应时延、峰值并发

### 4.3 验收标准

- **权限判定**：任意请求权限判定 ≤ 50ms
- **多群组合并**：正确执行并集逻辑
- **配额扣减**：准确归因到当前工作群组
- **统计报表**：支持 5 个维度交叉查询

## 5. 系统架构与职责边界

### 5.1 背景与目标

明确前端、统一网关、后端编排三层职责边界，确保开发分工清晰。

### 5.2 系统分层

**三层架构**：

- **前端层（AgentifUI）**
  - 职责：交互渲染、状态展示、基础校验、用户反馈

- **统一接口网关**
  - 职责：身份鉴权、Trace 生成、协议适配、治理执行、降级

- **后端编排平台**
  - 支持：Dify / Coze / n8n / LangChain 等

### 5.3 职责分工

**用户身份鉴权**：
- 前端：发起请求
- 网关：**执行验证**
- 后端：-

**Trace ID 生成**：
- 前端：-
- 网关：**生成并贯穿**
- 后端：接收并传递

**权限判定**：
- 前端：展示结果
- 网关：**执行判定**
- 后端：-

**配额扣减**：
- 前端：-
- 网关：**执行扣减**
- 后端：-

**内容合规检测**：
- 前端：提示
- 网关：**执行检测**
- 后端：-

**流式响应**：
- 前端：**渲染**
- 网关：透传
- 后端：生成

**停止生成**：
- 前端：发起信号
- 网关：**转发/软停止**
- 后端：执行终止

**审计记录**：
- 前端：-
- 网关：**记录**
- 后端：-

### 5.4 统一接口网关能力

**核心职责**：
- 对前端提供兼容 OpenAI API 规范的统一调用方式
- 请求参数、响应结构、流式返回、错误处理一致性

**兼容范围（v1.0）**：
- 最小可用集合：对话 + 流式 + 最小工具调用
- OpenAI 全量能力不在承诺范围

**会话主键归属**：
- 会话主键由 AgentifUI 统一生成与管理
- 后端平台会话/执行标识作为映射信息维护
- 不承诺跨平台可续聊迁移

### 5.5 能力降级策略（v1.0 必须覆盖）

- **不支持引用**：引用区隐藏，显示不可用提示
- **不支持 stop**：软停止（停止渲染）+ 提示"后端仍可能执行"
- **不支持工具调用**：降级为文本提示，结构化交互不可用说明
- **编排平台不可用**：保持登录/导航/历史/统计/审计可用，生成入口明确不可用提示

### 5.6 凭证与身份传递

- 前端 → 网关：平台登录态 + 身份声明
- 网关 → 后端：平台集成凭证，保证审计关联
- **默认不透传用户在后端的 access_token**

### 5.7 验收标准

- **降级覆盖**：4 类降级场景 100% 实现
- **协议兼容**：OpenAI 基础 API 兼容测试通过

## 6. 核心使用体验

### 6.1 背景与目标

为用户提供统一、流畅、一致的 AI 交互体验，屏蔽底层平台差异。

### 6.2 设计原则

- **统一体验**：对话、执行、状态反馈、错误提示保持一致
- **信息分层**：高频操作优先，低频能力收纳至二级入口
- **渐进式披露**：复杂能力按需展开
- **可治理追溯**：权限、配额、成本、安全相关操作清晰呈现

### 6.2a 应用发现与工作台

**应用展示**：
- 用户可见其有权访问的所有应用（基于授权并集）
- 支持分类浏览（按应用类型/用途标签）
- 支持搜索应用名称与描述

**工作台布局**：
- **最近使用**：按时间倒序，快速进入
- **我的收藏**：用户主动收藏的常用应用
- **全部应用**：可筛选、可搜索

**应用卡片信息**：应用图标、名称、简介、类型标签

### 6.3 对话与交互能力

**核心功能**：
- 实时文本对话
- 文件上传、下载与预览
- 流式响应展示（Streaming）
- 数学公式渲染（LaTeX）
- AI 推荐下一问题
- 停止生成（见 6.5）
- Artifacts 预览（见 6.6）
- Human-in-the-loop（见 6.7）

**对话管理**：
- 历史列表、搜索、重命名、删除、固定
- 对话项目/分组管理
- 归档与状态管理
- 消息复制、编辑重发、AI 回复重新生成
- 对话反馈（赞/踩/文本）

**对话分享**：
- 只读分享链接
- **可见范围与登录要求的关系**：
  - 启用登录（默认）：仅同 Tenant、同群组可见，需登录校验
  - 关闭登录：公开链接，任何人可访问，不限制范围
- 有效期：默认永久有效直至撤销，Tenant 可配置默认有效期与最长有效期（永久即 0，不受最长有效期限制）

### 6.4 文件上传

**边界**：仅用于对话/工作流输入与结果交付，不做网盘

**职责分工**：
- 前端：交互与发起上传、基础校验与提示
- 网关/后端：落库存储、病毒扫描、内容合规、PII 检测

**默认策略**：

- **单文件大小**：默认 50 MB，Tenant 可配 ≤ 200 MB
- **文件类型**：默认文档/图片/音视频白名单，可配置
- **单次数量**：默认 10 个，Tenant 可配 ≤ 20 个
- **保留期**：默认 90 天，Tenant 可配 30 天 ~ 永久

**配置层级**：Tenant 级默认值 → 应用级可覆盖（仅可收紧，不可放宽）

### 6.5 停止生成

**统一体验承诺**：提供"停止生成"，目标为节省成本与减少无效输出

**执行逻辑**：
1. **优先硬停止**：后端支持则真正取消并审计
2. **不支持则软停止**：停止渲染 + 明确提示"后端可能仍在执行"

**保证**：一致体验与可追溯审计

### 6.6 Artifacts

**定位**：将代码/文档/可视化从对话中抽离为可管理产物

**版本规则**：
- 每次输出自动生成草稿版本
- 用户显式保存生成稳定版本
- **草稿保留数量**：最近 10 个，超出自动清理
- 稳定版本留存受 Tenant 策略约束

**下载**：至少支持通用文本导出

**安全边界**：可执行内容预览以安全为先（受限/沙箱，Tenant 可控）

### 6.7 Human-in-the-loop

**协议**：
- 结构化交互必须来自后端结构化输出
- 前端仅渲染与回传选择
- 统一网关定义并适配一套 schema，前端只认一种协议

**覆盖范围**：确认/选择/审批/补充信息等受控交互，不扩展为低代码编排

### 6.8 会话与执行数据持久化

**策略**：
- AgentifUI **本地持久化**会话与执行数据
- 尽量与编排层数据保持一致
- 会话与执行数据**默认永久保存**

**回源触发条件**：
1. 本地数据缺失
2. 用户主动请求同步
3. 管理员触发全量校验

**应用删除后的数据处理**：
- 历史对话与执行记录：**只读保留**
- 状态标记：UI 明确标识"应用已删除/不可用"
- 审计要求：不得删除历史记录，直至达到租户级保留期限

**数据删除优先级**：

| 删除操作 | 用户数据 | 审计日志 | 生效时机 |
|---------|---------|---------|---------|
| 用户自删对话 | ✅ 可删 | ❌ 保留删除事件 | 即时 |
| Admin 删除用户 | ⚠️ 脱敏保留 | ❌ 保留 | 30 天冷静期后 |
| ROOT 删除租户 | ✅ 可删 | ⚠️ 保留至法务期限 | 即时标记，延迟清理 |

> **法务最短保留期**：审计日志即使租户删除，仍需保留至少 180 天或适用法规要求时间，以较长者为准。

**回源失败降级**：展示本地已有数据 + 提示"部分数据可能不完整"

### 6.9 验收标准

- **对话发送响应**：首 Token 时间 P95 ≤ 1.5s
- **文件上传**：50MB 文件上传 P95 ≤ 10s
- **停止生成**：用户点击后 ≤ 500ms 停止渲染

## 7. 运行与执行系统

### 7.1 背景与目标

提供统一的执行追踪与观测能力，支持多类型 Run 的状态管理与性能分析。

### 7.2 执行类型（Run Type）

- **Workflow Run**：工作流编排执行的运行实例
- **Agent Run**：智能体多轮自治与工具调用的运行实例
- **Generation Run**：单次或轻量交互的文本生成运行实例

**统一性**：三类 Run 提供一致的运行记录入口、状态展示与治理统计入口

### 7.3 执行状态

待处理 → 运行中 → 已完成 / 失败 / 已停止

**执行记录展示**：
- 执行步骤数
- Token 使用量
- 执行耗时
- 失败原因

### 7.4 Trace 与观测

**Trace 归属**：
- **Trace ID 由统一接口网关生成并贯穿全链路**
- 前端仅负责展示与携带上游关联信息
- 格式兼容 OpenTelemetry/W3C traceparent

**外部观测平台**：
- 支持配置：全局默认 + Tenant 级覆盖
- 跳转方式：基于 Trace ID 的 URL 模板

**验收口径**：
- 任意 Run 的运行记录页面必须展示 Trace 标识
- 提供一键跳转入口
- 未配置外部观测平台时可复制 Trace 标识

### 7.5 验收标准

- **Trace 覆盖**：100% Run 记录包含 Trace ID
- **状态更新**：状态变更 ≤ 3s 反映到 UI

## 8. 安全与合规

### 8.1 背景与目标

为企业提供安全防护与合规保障能力，满足企业级审计与风控要求。

### 8.2 安全防护

**提示词注入检测**：
- 执行点：统一接口网关层
- 前端仅提示与引导，不作为安全边界
- 检测结果支持：记录、告警、拦截
- **默认**：记录 + 告警，不拦截；Tenant 可配置拦截

**会话隔离**：
- 用户隔离：不同用户会话默认不可见
- 应用隔离：不同应用上下文默认不互通
- v1.0 **不启用**跨应用引用历史/统一记忆

**访问控制**：按权限模型执行

### 8.3 内容合规

**PII 去敏**：
- Tenant 全局 / 应用级开关
- 默认作用于：日志与导出等落盘与扩散环节
- 用户输入与模型输出：按业务配置
- **v1.0 仅做不可还原去敏**，关联追溯为 v2.0+

**输出合规**：
- 支持 Tenant 级合规策略配置
- AI 输出内容合规检测与拦截

### 8.4 审计覆盖范围

**默认审计事件类别**（关键安全与治理事件不可关闭）：

- **鉴权事件**：登录/登出/SSO、失败登录、token 发放/刷新/撤销
- **权限变更**：角色、群组成员、应用授权、ACL、配额、合规策略
- **高风险访问**：查看/导出他人对话、break-glass 操作
- **治理安全**：拦截、脱敏、告警、策略命中
- **管理后台**：创建/删除/禁用 Tenant/App/用户/群组

**通用字段**：操作者、时间、对象、动作、结果、来源 IP/UA、事由、Trace ID

### 8.5 审计日志不可篡改

**v1.0 最低标准**：
- **逻辑不可篡改**：产品层面不可编辑、不可删除
- 按权限严格控制可见与导出
- 关键操作可追溯

**保留期**：
- 默认：180 天
- Tenant 可配：最长 7 年

**扩展预留**：对接外部审计存储/追加写等更强不可篡改能力

### 8.6 验收标准

- **审计覆盖**：5 类事件 100% 覆盖
- **审计延迟**：事件发生后 ≤ 5s 可查询
- **日志保留**：≥ 180 天
- **PII 去敏**：敏感字段检出率 ≥ 95%

## 11. 性能与高可用

### 11.1 规模指标

- **单 Tenant 用户规模**：≥ 5 万用户
- **同时在线**：≥ 1,000
- **活跃交互并发**：≥ 300
- **流式连接并发**：≥ 500

### 11.2 延迟指标

- **页面交互延迟**（分级）：
  - **简单交互**（按钮/弹窗/表单）：P95 ≤ 300ms
  - **复杂列表**（分页/过滤/排序）：P95 ≤ 800ms
- **对话首 Token 时间（TTFT）**：P95 ≤ 1.5s，P99 ≤ 3s
- **AgentifUI 自身新增开销**：P95 ≤ 200ms，P99 ≤ 400ms
- **配额告警延迟**：≤ 5 分钟

### 11.3 可用性目标

- **平台月度可用性**：≥ 99.9%
- **外部编排不可用时**：只读能力可用（历史/统计/审计）

### 11.4 高可用与降级

**分层责任**：
- AgentifUI 侧：前端与统一网关 HA
- 对接方：外部编排平台/模型侧可用性
- AgentifUI 负责：可解释错误与可观测追踪

**无单点要求**：
- 前端与网关均支持多副本部署与滚动升级
- 任一实例故障不影响核心链路

**降级策略**：
- 外部编排平台故障时保持：登录、导航、历史、统计、审计可用
- 生成/执行入口给出明确不可用提示与追踪信息

### 11.5 数据备份与恢复策略

- **平台级备份**：
  - **策略**：每日全量备份（凌晨 02:00 UTC）
  - **保留期**：近 7 天
  - **恢复目标**：RPO ≤ 24h，RTO ≤ 4h
- **租户级数据**：
  - 支持租户管理员手动导出关键业务数据（对话/审计/配置）用于本地归档

  - 定义：平台级运维角色，作用域为 Platform，负责租户生命周期管理

- **Tenant Admin**
  - 定义：租户级最高治理权限角色，作用域为 Tenant

- **Manager**
  - 定义：群组管理角色，作用域为 Group，非独立实体而是角色绑定

- **Break-glass**
  - 定义：紧急访问机制，仅限运维排障时使用，需强制留痕与审计

- **Artifacts**
  - 定义：从对话中抽离的可管理产物，如代码、文档、可视化内容

- **PII (Personally Identifiable Information)**
  - 定义：个人可识别信息，平台支持去敏处理

- **MFA (Multi-Factor Authentication)**
  - 定义：多因素认证，v1.0 支持 TOTP 方式

- **Trace ID**
  - 定义：全链路追踪标识，由统一网关生成，兼容 OpenTelemetry/W3C traceparent
